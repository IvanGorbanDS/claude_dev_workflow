## Verdict: REVISE

## Summary

The plan is well-organized and shows clear lineage to the architecture (the architecture's recursion-bug risk is explicitly mitigated, the fixture-vs-block divergence is pre-emptively documented in the plan's risk on fixture drift, and lesson 2026-04-23 LLM-replay non-determinism is honored in the plan's structural-test-only decision). However, three load-bearing assumptions are unverified: (a) the Section-zero prose nowhere defines what "Agent subagent" tool actually is in this Claude Code harness, while existing prose in `architect/SKILL.md:104` already warns that "model specification is not supported in the current harness version" is a real failure mode; (b) the recursion guard described in the plan does NOT match the architecture's I-02 contract — the plan changes the parent's prepended sentinel from bare to counter form `[no-redispatch:1]`, and the abort fires at counter value 2 or higher, but the structural test's assertion contract is silent on this drift; and (c) the test asserting the Section-zero block contains the literal substring `model: "haiku"` / `model: "sonnet"` will collide with the literal `model: sonnet`/`model: haiku` strings in the SKILL.md YAML frontmatter, invalidating the test's discrimination claim.

## Issues

### Critical (blocks implementation)

- **Issue C1 — Section-zero prose references an "Agent subagent" mechanism whose existence and model parameter is unverified in this harness**
  - What: The plan's first task (the fixture-author task at plan lines 41–43) specifies the Section-zero dispatch action as "Spawn an Agent subagent with: model: declared / description: skill name dispatched at declared tier / prompt: ...". No task in this plan verifies that the harness actually exposes such a tool with a model parameter that switches tiers. The architecture's own scan-agent precedent at `dev-workflow/skills/architect/SKILL.md:104` documents the failure mode explicitly: "If model specification is not supported in the current harness version, the scan agents still provide value through structured extraction and context isolation, though the model-tiering cost savings would not apply." For the architect's scan agents this is graceful degradation (Sonnet vs Opus on a read-only summarization task). For Section-zero dispatch it is a SILENT FAILURE OF THE COST GUARDRAIL — every cheap-tier invocation will continue running on Opus 1M while the prose claims it dispatched, and the user has no way to detect this from an outside observation. The plan's manual-smoke task tries to verify by looking for "Spawning subagent..." in the harness output, but that's empirical-after-merge, not pre-merge.
  - Why it matters: The entire stage's ROI premise (architecture line 91: "for /end_of_day's structured-template work … Opus 1M → Haiku is ~30x cost reduction") evaporates if dispatch silently no-ops. Worse, the recursion guard in the Section-zero block only fires when dispatch IS happening — so a fail-open dispatch silently-no-ops without ever testing the abort branch in production.
  - Where: plan task at lines 41–43; manual-smoke procedure at lines 246–251; architecture I-01 (line 303) anticipated this with "Default to no-dispatch (proceed at current tier) on detection failure" but the plan never wires that detection.
  - Suggestion: Add a new task BEFORE the fixture-author task — "Empirically verify the harness exposes a subagent-spawn tool that accepts a model parameter and actually switches tiers." Concretely: in this very session (or a sibling diagnostic session) attempt the dispatch syntax once, measure whether the response shows the tier handoff, and DOCUMENT the exact tool name/parameter shape in the Section-zero prose. If the harness does not support model-switching subagents, the plan must either be deferred until it does, or shift to a different mechanism (e.g., the Skill tool, which the system prompt describes as running "within the main conversation" — meaning it would NOT switch tiers either). Per lesson 2026-04-25 set-u rc-hang, "Plan acceptance criteria that depend on shell-environment semantics need empirical verification on a realistic developer machine — not synthetic test fixtures" — the same lesson applies here: harness-tool semantics need empirical verification before locking the Section-zero wording to a specific tool name.

- **Issue C2 — The structural test's substitution-check assertion will false-pass via YAML-frontmatter collision and never test the Section-zero substitution**
  - What: The structural-consistency test case (c) at plan line 95 reads: "for each of the 12 files, parse YAML frontmatter to extract `model:` value (must be `haiku` or `sonnet`), then assert the Section-zero block in that file contains the literal substring `model: "<that value>"` (the dispatch instruction's `model:` line). NO file should contain the literal placeholder after substitution." Two problems compound: (i) the SKILL.md YAML frontmatter ALREADY contains `model: sonnet` or `model: haiku` (verified via grep on `dev-workflow/skills/revise/SKILL.md` returning `4:model: opus` and similar for all skills). The test as written checks "the Section-zero block contains `model: <value>`" — but the test would have to first slice the Section-zero block out of the file to avoid colliding with the frontmatter; the plan does not specify the slicing logic. (ii) The Section-zero prose template in the fixture-author task never actually writes the line `model: "haiku"` — it uses `model: declared` as a placeholder. The plan does not define what the post-substitution line literally looks like in the Section-zero block. So the test asserts presence of a string that the Section-zero template may or may not produce.
  - Why it matters: A test that false-passes (matches the frontmatter `model: sonnet` / `model: haiku` instead of the Section-zero dispatch instruction) leaves the dispatch-to-non-existent-model risk completely unguarded. Per lesson 2026-04-24 fixture filename auto-detection ("A test that 'passes' because it never exercised the code path is worse than a failing test"), this exact failure pattern just bit Stage 4 of artifact-format-architecture; the plan must not repeat it.
  - Where: fixture-author task at plan lines 36–49 (template never specifies the post-substitution literal); structural-consistency test at plan line 95 (assertion collides with frontmatter).
  - Suggestion: (a) In the fixture-author task, write the Section-zero template's dispatch line in a quotation form that is GUARANTEED to be unique across the file — e.g., use an XML-tag wrapping per format-kit §1 primitive 6, or use the verbatim phrase `dispatched-tier: haiku` that does not appear in YAML frontmatter. (b) In the structural-consistency test, rewrite case (c) to first slice the Section-zero block (between the Section-zero H2 heading and the next H2 heading), then assert the substitution within that slice ONLY. Add an explicit negative assertion: "outside the Section-zero slice, the test does not consult `model:` strings." (c) Add a new test case that verifies the post-substitution literal matches a fixture string exactly — this catches the dispatch-to-nonexistent-model risk deterministically.

- **Issue C3 — Recursion-guard contract drifts between architecture I-02 and the fixture-author task without a passing-direction test**
  - What: Architecture line 88 says: "if an Agent subagent re-loads the same skill, the prompt already has the sentinel, so it skips dispatch and runs the body" — i.e., the parent prepends bare `[no-redispatch]` and the child sees it and proceeds. Architecture I-02 (line 304) adds: "Hard cap: detect 2nd dispatch attempt (sentinel-counter via Agent prompt convention `[no-redispatch:N]`); abort w/ error if N≥2." Plan fixture-author sub-bullet (c) at line 42 changes the contract: "When dispatching from the parent (no sentinel present), prefix child prompt with `[no-redispatch:1]`." So in the plan's design, EVERY dispatch produces `[no-redispatch:1]`, never bare `[no-redispatch]`. The bare form is reserved for "user manually types it" (plan task sub-bullet d). The structural-consistency test case (e) at plan line 97 asserts both literal substrings are present in the Section-zero prose, but does NOT assert which one the dispatch action emits. The synthetic-recursion test case (d) at plan line 114 requires "the bare token `[no-redispatch]` count ≥ 2 (one for parent's prepend rule, one for the user's manual override)" — but per the fixture-author sub-bullet (c), the parent prepends `[no-redispatch:1]`, NOT bare `[no-redispatch]`, so the "parent's prepend rule" reference is a contradiction.
  - Why it matters: This is exactly the V-07 prefix-match issue from lesson 2026-04-25 ("Validator invariants and writer formats must be designed together"). The Section-zero prose, the parent-emit rule, the child-detect rule, the abort rule, and the test-assertion rule are all designed independently and don't compose. In production, a child seeing `[no-redispatch:1]` checks "is counter ≥ 2? no → proceed to Section-one." Fine. But what if a parent in a future skill update accidentally emits bare `[no-redispatch]` (no counter)? The child's detect rule per the fixture-author sub-bullet (a) accepts bare form ALSO; it proceeds to Section-one; the abort branch never fires. The counter is only useful for catching a buggy parent that emits `[no-redispatch:2]` — which no parent in the design ever should. So the counter as designed only catches manual user error (`[no-redispatch:2] /gate`), not the recursion bug the plan's risk row claims it catches.
  - Where: architecture line 88 vs plan task at line 42; synthetic-recursion test case (d) at plan line 114; the recursion-bug risk-row mitigation column at plan line 198.
  - Suggestion: Either (a) revise the contract to make the counter monotonically increment — every dispatch reads incoming counter (default 0 if bare or absent), increments to N+1, prepends `[no-redispatch:N+1]` to the child. Abort if incoming counter is 2 or higher. This makes the counter a real recursion detector. Document in the fixture-author task with a worked example (parent: no sentinel → child gets `:1`; child re-dispatches: gets `:2` → ABORT). (b) OR: drop the counter form entirely from the plan and use bare-sentinel-only (matching the architecture's primary description). Either way, the structural-consistency test and the synthetic-recursion test must be rewritten so the assertion set matches the chosen contract — not the current half-of-each. (c) Add an explicit unit test (in the synthetic-recursion test): "given a Section-zero block, simulate a child seeing `[no-redispatch:2]` and assert the documented branch is the abort branch" — pure string matching is fine here, just make the contract testable end-to-end.

- **Issue M3 — The manual-smoke task is the SOLE test for production dispatch behavior; it is HUMAN-IN-THE-LOOP and gated post-implement**
  - What: The manual-smoke task (plan lines 158–171) is explicitly HITL ("cannot be fully automated because it depends on Claude Code's Agent tool dispatch behavior in production"). All structural tests (the structural-consistency test and the synthetic-recursion test) only verify that the Section-zero prose CONTAINS the right strings — they do not verify the dispatch ACTUALLY HAPPENS. The manual-smoke task happens after `/implement` finishes and after `bash install.sh` redeploys, which means the user discovers a broken dispatch only after twelve files are modified, committed, and a fresh install.sh has overwritten ~/.claude/skills.
  - Why it matters: Per the architecture's recursion-bug risk row (Likelihood Low, Impact High), the safety property hinges on dispatch behaving as designed. A purely-structural-test stage-1 ships a prose-only feature — and the feature flag is the install.sh redeploy (which is not reversible without running `git revert` and re-running `install.sh`). Lesson 2026-04-23 (LLM-replay gates non-deterministic) correctly forbids LLM replay in CI; the plan's test-design decision row acknowledges this. But there is a middle path: a tiny SDK-call POC test (NOT integrated with CI; run by the user once at install) that calls the actual harness's subagent-spawn tool with `model: "haiku"` and prints the tier of the response. That single POC would empirically validate Issue C1 and de-risk the manual-smoke task.
  - Suggestion: Add a new task (after Issue C1 is resolved): "Author a one-shot diagnostic script `dev-workflow/scripts/verify_subagent_dispatch.py` that the user runs ONCE post-install to confirm subagent-spawn tool works with model parameter. Acceptance: script prints `[verified: subagent dispatch reaches model X]` or `[failed: harness does not support model parameter]`. The Section-zero dispatch behavior is gated on this verification." Tie the manual-smoke task's pass/fail criterion to this script's output, not to the user's eyeball reading of the harness banner.

- **Issue M6 — The `test_detection_rule_consistency` byte-equality mitigation claim is incomplete**
  - What: The byte-equality risk-row mitigation column (plan line 199) claims: "the §5.7.1 block stays verbatim and in its original position in the file's body; only its line number shifts (which the byte-equality test is robust to — it tests substring presence, not absolute line position)". - **Issue M7 — The structural-consistency and synthetic-recursion tests overlap; the ordering assertion does not survive trivial reordering by future skill authors**
  - What: The structural-consistency test case (a) asserts "literal heading appears EXACTLY ONCE."   - Suggestion: (a) Drop the redundant case from the synthetic-recursion test (or fold it into the consistency test with a `pytest.mark.smoke` decoration to make the redundancy intentional).
