
## 2026-04-13 — run-skill
**What happened:** When adding a skill that orchestrates other skills, the critic caught that the target skills (`implement/SKILL.md`, `end_of_task/SKILL.md`) had their own hard rules against auto-invocation. Without exception clauses in those files, subagents spawned by `/run` would have read their own SKILL.md and refused to execute.
**Lesson:** When any skill changes the invocation rules for another skill (e.g., adding an exception, extending permissions), update the *target* skill's SKILL.md too — not just CLAUDE.md. The subagent reads its own SKILL.md, not CLAUDE.md.
**Applies to:** /plan, /critic — watch for cross-skill permission changes in orchestrator designs.

## 2026-04-18 — memory-cache (Stages 0–2)
**What happened:** Stages 0, 1, and 2 all shared the same task folder (`.workflow_artifacts/memory-cache/`), so `current-plan.md` got overwritten each stage and critic responses from different stages mixed together (e.g., `critic-response-1.md` from Stage 0 vs Stage 1 are indistinguishable by filename).
**Lesson:** When a task has multiple stages, create a subfolder per stage inside the task folder (e.g., `.workflow_artifacts/memory-cache/stage-2/current-plan.md`). This keeps plans, critic responses, and reviews cleanly separated and preserves the history of each stage without relying on git.
**Applies to:** /thorough_plan, /plan, /critic, /revise, /review — any skill that writes artifacts to the task folder should use the stage subfolder when the task has multiple stages.

## 2026-04-20 — memory-cache Stage 7 benchmark
**What happened:** First post-cache benchmark compared baseline (Opus 4.6, pre-Stage-1 codebase) vs post-cache (Opus 4.7, post-Stage-6 codebase) and showed a 172% token regression. It looked like the cache was useless. A clean single-variable rerun (cache-off vs cache-on, both Opus 4.7 on current main) instead showed the cache saves 47% tokens and 43% cost per lifecycle.
**Lesson:** Validation benchmarks must hold all confounders constant. Model version and codebase size are the two biggest — a simultaneous model upgrade + codebase growth completely masked the cache's benefit in the first comparison. When the headline number is surprising, check the confounders before accepting the conclusion. Also: the cache's dominant mechanism was not "reducing re-reads" (Anthropic's prompt cache already did that at 91%+ hit rate) — it was **preventing wasteful exploratory reading**. Without `_index.md`, the model fell back to reading README, SETUP, QUICKSTART, Workflow-User-Guide, and tangential SKILL.md files trying to orient itself.
**Applies to:** /architect, /plan, /critic, /review — when benchmarking or reasoning about cost savings, always check for confounders; before/after comparisons across codebase revisions or model versions are not apples-to-apples.

## 2026-04-22 — critic-on-architecture (cost aggregation is broken)
**What happened:** `/end_of_task` reported "cost unknown" because `ccusage` is not installed on this machine. The cost ledger tracked UUIDs + phases faithfully but never stored dollar amounts; without `ccusage`, the ledger is purely descriptive. Manual JSONL scan across 2026-04-21/22 sessions showed the task actually cost **~$140** (Opus $128.71, Sonnet $11.19, Haiku $0.41), vs. the CLAUDE.md Medium-profile estimate of $2.99–$4.00 — ~35× over.
**Lesson:** The cost-tracking system is half-built. `/end_of_task` must have a JSONL-parse fallback that computes `input_tokens × price + output_tokens × price + cache_* × price` per `message.model` directly from `~/.claude/projects/<project>/*.jsonl`, scoped to the task's date range. Current ledger-only output is theater. Also: the CLAUDE.md Medium cost band is off by more than an order of magnitude for tasks that trigger multiple critic/review rounds — the band should be revised or clearly flagged as "happy path only."
**Applies to:** /end_of_task, /cost_snapshot, /thorough_plan — ship a fallback cost computer; re-baseline the profile cost bands from real task history; consider surfacing running cost inline during long loops so the user can intervene mid-flight.

## 2026-04-22 — critic-on-architecture (fixture-first traps when baselines rot)
**What happened:** Task plan's Risk R1 tripwire used "reference fingerprints" captured from pre-revision archived `critic-response-1.md` files as the TC1/TC2 baseline, committed fixture-first before any code change. Post-change `/critic` runs against post-revision plans legitimately found fewer issues (because earlier revisions fixed them), which made TC1 "fail" — not a regression, just an obsolete baseline. Cost: review round 1 CRIT-1, full TC1/TC2 redesign into functional-contract + reproducibility tests, re-execution, review round 2.
**Lesson:** Fixture-first ordering only works when the fixture represents a stable contract (API shape, wire format, output structure). Never use historical LLM output as a regression baseline when the upstream input is itself being revised — the baseline will desync the moment the input changes, and the test will fire for the wrong reason. When tempted to capture an LLM output as a fixture, ask: "Is this a structural contract or just a frozen historical artifact?" If the latter, write a contract test instead.
**Applies to:** /plan, /critic — when a plan proposes fixture-based regression tests, explicitly verify the fixture source is stable under revision; flag fingerprint-of-prior-LLM-output fixtures as high-risk.

## 2026-04-22 — critic-on-architecture (Opus-always critic is disproportionate for docs-only changes)
**What happened:** The task was a markdown/docs-only change (11 files, 546 lines, zero source code). Every `/critic` round still ran on Opus per CLAUDE.md's non-negotiable "critic always Opus" rule. Combined with the recursive self-critique (critiquing a plan that adds a new critic target), three fresh-session critics, two fresh-session reviews, and ~37M cache-read tokens at $1.50/M, Opus alone consumed ~$128 of the ~$140 total. Round-3 targeted edit for MAJ-8 (cosmetic sentinel-file framing) was over-engineering.
**Lesson:** "Critic always Opus" is a safety rule that currently ignores the cost/benefit ratio on docs-only or rubric-only changes. Worth considering: (a) a `docs:` profile tag that routes `/critic` to Sonnet with a reduced rubric focused on consistency/clarity rather than integration safety; (b) explicit scope discipline — cosmetic framing fixes (MAJ-8 style) roll into `/implement`, not a separate revise round; (c) detection of "recursive self-critique" (task modifies `/critic` itself, `/architect` itself, etc.) and a warning that each round re-reads the skill-being-modified.
**Applies to:** /thorough_plan, /critic — revisit the always-Opus rule for docs/rubric changes; add a "cosmetic framing → roll into implement, don't revise" guideline; flag recursive self-critique at the triage step.

## 2026-04-23 — caveman-token-optimization Stage 1 (LLM-replay gates are non-deterministic and expensive)
**What happened:** Original Task 12 dollar gate used LLM replay of `/implement` on synthetic fixtures. Critic CRIT-2 called this out as non-deterministic, expensive (~$15–30 per gate run), and dependent on interactive human prompting (~30 prompts). Revision replaced it with a deterministic structural cost proxy: `(S_pre_mean − S_post_mean) × tokens_per_byte × cache_read_price × est_reads_per_task`.
**Lesson:** When LLM replay would make a gate non-deterministic and expensive, replace the replay with a deterministic proxy over measurable quantities (artifact byte-sizes, documented constants). The proxy's assumptions become auditable, and the gate signal arrives early. A proxy FAIL is an equally valid signal — it tells you before shipping whether the design suffices.
**Applies to:** /plan, /critic, /architect

## 2026-04-23 — caveman-token-optimization Stage 1 (rubric refs belong at the write-site, not in a preamble)
**What happened:** Task 3 rubric references were required to go inside the relevant write-instruction section of each SKILL.md, NOT in a generic "About" or "Process" preamble. If placed too high, the model reads the reference but doesn't apply it to the right artifact (it may apply it to unrelated outputs or to Tier 1 files).
**Lesson:** Rubric references must be placed immediately before or inside the write-to-disk instruction for the named artifact. Grep the anchor section after adding to verify placement (5 lines of context before and after). Placement above the write site is diff noise at best, regression at worst.
**Applies to:** /implement, /review

## 2026-04-24 — artifact-format-architecture Stage 2 (Class B writer wiring)
**What happened:** The integration test for `test_integration_current_plan_body` initially asserted Haiku must NOT emit a `## For human` heading in its stdout. Live Haiku calls showed it emits the heading inconsistently (3 of 5 samples). The test had to be relaxed to mirror §5.3 Step 3(a) dedup behavior: strip a leading `## For human` before counting non-blank summary lines.
**Lesson:** When testing a script whose contract is "the caller handles the heading", the test must mirror the caller's behavior — not assert a stricter precondition. ## 2026-04-24 — artifact-format-architecture Stage 2 (fixture filename auto-detection)
**What happened:** T-10 boundary fixture files were initially named `v3-current-plan-summary-12.md` and `v3-current-plan-summary-13.md`. `validate_artifact.py`'s `detect_type()` uses `re.match(r'^current-plan', name)` — the `v3-` prefix broke auto-detection, so V-06 never fired on those fixtures. ## 2026-04-25 — artifact-format-architecture Stage 4 (V-07 prefix-match relaxation discovered during fixture authoring)
**What happened:** During Stage 4 fixture-test authoring, the v3 critic-response fixture failed V-07 because the format-kit canonical heading-line-form is `## Verdict: PASS | REVISE`, but V-07 required exact match of `## Verdict`. The fix relaxed V-07 to prefix-match (`## Verdict: PASS` satisfies required `## Verdict`). This was an unplanned semantic change to a validator invariant, discovered empirically rather than at planning time. Round-1 review flagged it as MAJOR governance, recommended a dedicated unit test before merge.
**Lesson:** Validator invariants and the writer formats they enforce must be designed together. When format-kit specifies a heading-line-form (heading + value on the same line, like `## Verdict: PASS`), the corresponding required-section check needs explicit prefix-match semantics from day one. Reviewers/critics should ask "does this writer format pass V-07 as written?" against every section labeled REQUIRED in format-kit.sections.json.
**Applies to:** /architect, /plan, /critic — flag invariant-vs-writer-format misalignment at planning time, not implementation time.

## 2026-04-25 — artifact-format-architecture Stage 4 (`set -u` is too strict for sourcing user rc files)
**What happened:** The Stage 4 with_env.sh wrapper used `set -eu` and sourced the user's `~/.zshrc` to load ANTHROPIC_API_KEY for non-interactive subshells. Real-world rc files commonly reference unset variables (`$NVM_DIR`, theme vars, plugin-manager guards), and `set -u` aborts the wrapper before it can exec the real command. The Stage 4 smoke documented this as a deterministic hang and used a manual workaround (`bash -c '. ~/.zshrc 2>/dev/null; export ANTHROPIC_API_KEY; python3 ...'`) at every Class B Haiku call. Production users would have hit silent v2-fallback. Fixed post-review by dropping `set -u` (keep `set -e`) and removing dead-code subshell-sourcing lines, plus a regression test using a synthetic rc file with unset-var references.
**Lesson:** Plan acceptance criteria that depend on shell-environment semantics (env loading, rc file sourcing, non-interactive subshells) need empirical verification on a realistic developer machine — not synthetic test fixtures with clean rc files. When designing a wrapper that sources user shell config, default to `set -e` only (or scope `set -u` to the wrapper's own logic with `set +u` around the source). Add a regression test with an intentionally messy rc file (unset-var references, plugin guards) before shipping.
**Applies to:** /architect, /plan, /critic, /implement — for any task that wraps user-shell behavior, add "test against a messy rc file" to the acceptance checklist.
