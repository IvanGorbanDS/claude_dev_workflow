Development Workflow

User Guide

How to work with /discover, /architect, /thorough_plan, /gate, /implement, /review, /end_of_task, /rollback, /start_of_day, /end_of_day

Quick Reference

Your workflow has 13 commands in three groups: the development pipeline, safety controls, and session management.

The Pipeline

CommandWhat it does
/init_workflowOne-time project bootstrap. Creates directory structure, copies all skills, runs /discover, generates quickstart guide.
/discoverScans all repos, builds inventory, architecture map, and git log. Run once on setup.
/architectDeep exploration with Opus. Reads repos, docs, web. Produces architecture.md with decomposed stages.
/thorough_planOrchestrates /plan → /critic → /revise loop (up to 5 rounds) until convergence.
/planCreates the initial detailed plan. Reads lessons-learned.md. Can be used standalone.
/criticReviews plan in a fresh session against actual code. Reads lessons-learned.md.
/reviseAddresses critic feedback. Updates current-plan.md.
/implementWrites code from the plan. Requires your explicit command. Never auto-invoked.
/reviewOpus verifies implementation vs plan. Integration safety, risk assessment.
/end_of_taskFinalizes accepted work. Commits, pushes branch to remote, captures lessons. Does NOT create PR — you do that separately. Requires your explicit command.

Safety Controls

CommandWhat it does
/gateRuns automated checks (tests, lint, secrets scan) and STOPS for your approval between every phase.
/rollbackMaps commits to plan tasks. Shows what would revert. Requires your confirmation before acting.

Session Management

CommandWhat it does
/start_of_dayReads daily cache, checks git state, presents briefing with exactly where to resume.
/end_of_daySaves session state, updates git log, consolidates unfinished work, prompts for lessons learned.

Key Rules to Remember

1. /implement and /end_of_task never run automatically. You must type them. Claude stops after planning converges (waiting for /implement) and after review approves (waiting for /end_of_task). Nothing gets written or shipped without your conscious decision.
2. Gates run between every phase. They run automated checks and present results. You approve before the next phase starts. Even if all checks pass, Claude waits.
3. /critic always runs in a fresh session. This prevents bias from having just written the plan. It reads the actual code, not just the plan's claims about it.
4. Lessons learned accumulate. /end_of_day prompts you for insights. /plan and /critic read them at the start. Over time, the workflow gets smarter about your specific codebase and patterns.
5. You can skip stages. Small bug fix? Go straight to /plan → /implement → /review → /end_of_task. The full pipeline (/discover → /architect → /thorough_plan) is for large features.
6. /rollback is always available. It maps commits to plan tasks, shows what will change, and requires your confirmation. Safe to use at any point.
7. Each task gets its own folder. refund-flow/, auth-token-fix/ — parallel tasks never mix. Descriptive kebab-case names.

Session Management: One Command Per Session

Context windows fill up. This is expected and the workflow is designed for it. Each command can run in its own chat session — the files on disk are the shared memory between sessions.

How it works

Every skill has a "session bootstrap" step. When it starts in a fresh session, it reads what it needs from disk:

SkillReads on startup
/architectCLAUDE.md, lessons-learned.md, /discover memory files, task subfolder
/planCLAUDE.md, lessons-learned.md, architecture.md, prior critic responses
/criticCLAUDE.md, lessons-learned.md, current-plan.md, actual source code
/reviseCLAUDE.md, current-plan.md, latest critic-response
/implementCLAUDE.md, lessons-learned.md, session state, current-plan.md, source code
/reviewCLAUDE.md, lessons-learned.md, current-plan.md, architecture.md, git diff + full files

You never have to re-explain context. If it's in the files, the skill reads it.

Recommended session pattern

Large feature (multi-session)

Session 1: /architect — design the refund flow

Claude saves architecture.md. Close this session.

Session 2: /thorough_plan — plan stage 1

Claude reads architecture.md from disk, runs plan→critic→revise. Saves current-plan.md. Close.

Session 3: /implement — tasks 1-4

Fresh context! Clean session means Claude focuses on code, not planning noise. Reads the plan from disk. Close when tasks are done.

Session 4: /review

Fresh eyes on the diff. Reads plan + code, no implementation bias. Close when done.

Small bug fix (single session)

Same session: /plan → /implement → /review

Short enough to fit in one context. No need to split.

Resuming mid-task

New session: /implement — continue with refund-flow
Reading session state… you completed tasks 1-4a last session. Tasks 5-8 remaining. Branch: feat/refund-flow. Picking up at Task 5.

The session state file tells the new session exactly where you left off.

When to split vs stay

Split when: you've been going for a while and responses are getting slower or less focused, you're moving to a different phase (plan→implement), or you want unbiased review.
Stay when: the task is small, you're in a quick plan→implement→review flow, or you're iterating on implementation fixes within the same phase.
Always split for: /critic (by design — it must be a fresh session), /review (fresh eyes catch more).
